6 // Currently we support two types of tables: plain table and block-based table.
7 // 1. Block-based table: this is the default table type that we inherited from
8 // LevelDB, which was designed for storing data in hard disk or flash
9 // device.
10 // 2. Plain table: it is one of RocksDB's SST file format optimized
11 // for low query latency on pure-memory or really low-latency media.
12 //

291  // Option to generate Bloom/Ribbon filters that minimize memory
292  // internal fragmentation.
293  //
294  // When false, malloc_usable_size is not available, or format_version < 5,
295  // filters are generated without regard to internal fragmentation when
296  // loaded into memory (historical behavior). When true (and
297  // malloc_usable_size is available and format_version >= 5), then
298  // filters are generated to "round up" and "round down" their sizes to
299  // minimize internal fragmentation when loaded into memory, assuming the
300  // reading DB has the same memory allocation characteristics as the
301  // generating DB. This option does not break forward or backward
302  // compatibility.
303  //
304  // While individual filters will vary in bits/key and false positive rate
305  // when setting is true, the implementation attempts to maintain a weighted
306  // average FP rate for filters consistent with this option set to false.
307  //
308  // With Jemalloc for example, this setting is expected to save about 10% of
309  // the memory footprint and block cache charge of filters, while increasing
310  // disk usage of filters by about 1-2% due to encoding efficiency losses
311  // with variance in bits/key.
312  //
313  // NOTE: Because some memory counted by block cache might be unmapped pages
314  // within internal fragmentation, this option can increase observed RSS
315  // memory usage. With cache_index_and_filter_blocks=true, this option makes
316  // the block cache better at using space it is allowed. (These issues
317  // should not arise with partitioned filters.)
318  //
319  // NOTE: Do not set to true if you do not trust malloc_usable_size. With
320  // this option, RocksDB might access an allocated memory object beyond its
321  // original size if malloc_usable_size says it is safe to do so. While this
322  // can be considered bad practice, it should not produce undefined behavior
323  // unless malloc_usable_size is buggy or broken.
324  bool optimize_filters_for_memory = false;

405  // This enum allows trading off increased index size for improved iterator
406  // seek performance in some situations, particularly when block cache is
407  // disabled (ReadOptions::fill_cache = false) and direct IO is
408  // enabled (DBOptions::use_direct_reads = true).
409  // The default mode is the best tradeoff for most use cases.
410  // This option only affects newly written tables.
411  //
412  // The index contains a key separating each pair of consecutive blocks.
413  // Let A be the highest key in one block, B the lowest key in the next block,
414  // and I the index entry separating these two blocks:
415  // [ ... A] I [B ...]
416  // I is allowed to be anywhere in [A, B).
417  // If an iterator is seeked to a key in (A, I], we'll unnecessarily read the
418  // first block, then immediately fall through to the second block.
419  // However, if I=A, this can't happen, and we'll read only the second block.
420  // In kNoShortening mode, we use I=A. In other modes, we use the shortest
421  // key in [A, B), which usually significantly reduces index size.
422  //
423  // There's a similar story for the last index entry, which is an upper bound
424  // of the highest key in the file. If it's shortened and therefore
425  // overestimated, iterator is likely to unnecessarily read the last data block
426  // from each file on each seek.
427  enum class IndexShorteningMode : char {
428  // Use full keys.
429  kNoShortening,
430  // Shorten index keys between blocks, but use full key for the last index
431  // key, which is the upper bound of the whole file.
432  kShortenSeparators,
433  // Shorten both keys between blocks and key after last block.
434  kShortenSeparatorsAndSuccessor,
435  };
436

440  // RocksDB does auto-readahead for iterators on noticing more than two reads
441  // for a table file if user doesn't provide readahead_size. The readahead
442  // starts at 8KB and doubles on every additional read upto
443  // max_auto_readahead_size and max_auto_readahead_size can be configured.
444  //
445  // Special Value: 0 - If max_auto_readahead_size is set 0 then no implicit
446  // auto prefetching will be done. If max_auto_readahead_size provided is less
447  // than 8KB (which is initial readahead size used by rocksdb in case of
448  // auto-readahead), readahead size will remain same as
449  // max_auto_readahead_size.
450  //
451  // Value should be provided along with KB i.e. 256 * 1024 as it will prefetch
452  // the blocks.
453  //
454  // Found that 256 KB readahead size provides the best performance, based on
455  // experiments, for auto readahead. Experiment data is in PR #3282.
456  //
457  // This parameter can be changed dynamically by
458  // DB::SetOptions({{"block_based_table_factory",
459  // "{max_auto_readahead_size=0;}"}}));
460  //
461  // Changing the value dynamically will only affect files opened after the
462  // change.
463  //
464  // Default: 256 KB (256 * 1024).
465  size_t max_auto_readahead_size = 256 * 1024;

467  // If enabled, prepopulate warm/hot blocks (data, uncompressed dict, index and
