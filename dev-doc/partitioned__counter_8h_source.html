56 // Overview: A partitioned_counter provides a counter that can be incremented
57 // and the running sum can be read at any time.
58 // We assume that increments are frequent, whereas reading is infrequent.
59 // Implementation hint: Use thread-local storage so each thread increments its
60 // own data. The increment does not require a lock or atomic operation.
61 // Reading the data can be performed by iterating over the thread-local
62 // versions, summing them up. The data structure also includes a sum for all
63 // the threads that have died. Use a pthread_key to create the thread-local
64 // versions. When a thread finishes, the system calls pthread_key destructor
65 // which can add that thread's copy into the sum_of_dead counter.
66 // Rationale: For statistics such as are found in engine status, we need a
67 // counter that requires no cache misses to increment. We've seen significant
68 // performance speedups by removing certain counters. Rather than removing
69 // those statistics, we would like to just make the counter fast. We generally
70 // increment the counters frequently, and want to fetch the values
71 // infrequently. The counters are monotonic. The counters can be split into
72 // many counters, which can be summed up at the end. We don't care if we get
73 // slightly out-of-date counter sums when we read the counter. We don't care
74 // if there is a race on reading the a counter
75 // variable and incrementing.
76 // See tests/test_partitioned_counter.c for some performance measurements.
77 // Operations:
78 // create_partitioned_counter Create a counter initialized to zero.
79 // destroy_partitioned_counter Destroy it.
80 // increment_partitioned_counter Increment it. This is the frequent
81 // operation. read_partitioned_counter Get the current value. This is
82 // infrequent.
83 // See partitioned_counter.cc for the abstraction function and representation
84 // invariant.
85 //
86 // The google style guide says to avoid using constructors, and it appears that
87 // constructors may have broken all the tests, because they called
88 // pthread_key_create before the key was actually created. So the google style
89 // guide may have some wisdom there...
90 //
91 // This version does not use constructors, essentially reverrting to the google
92 // C++ style guide.
93 //
94
