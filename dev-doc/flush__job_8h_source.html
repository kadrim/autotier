105  // Memtable Garbage Collection algorithm: a MemPurge takes the list
106  // of immutable memtables and filters out (or "purge") the outdated bytes
107  // out of it. The output (the filtered bytes, or "useful payload") is
108  // then transfered into a new memtable. If this memtable is filled, then
109  // the mempurge is aborted and rerouted to a regular flush process. Else,
110  // depending on the heuristics, placed onto the immutable memtable list.
111  // The addition to the imm list will not trigger a flush operation. The
112  // flush of the imm list will instead be triggered once the mutable memtable
113  // is added to the imm list.
114  // This process is typically intended for workloads with heavy overwrites
115  // when we want to avoid SSD writes (and reads) as much as possible.
116  // "MemPurge" is an experimental feature still at a very early stage
117  // of development. At the moment it is only compatible with the Get, Put,
118  // Delete operations as well as Iterators and CompactionFilters.
119  // For this early version, "MemPurge" is called by setting the
120  // options.experimental_mempurge_threshold value as >0.0. When this is
121  // the case, ALL automatic flush operations (kWRiteBufferManagerFull) will
122  // first go through the MemPurge process. Therefore, we strongly
123  // recommend all users not to set this flag as true given that the MemPurge
124  // process has not matured yet.
